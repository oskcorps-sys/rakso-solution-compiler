# RAKSO Solution Compiler Doctrine

## Mission

Convert business chaos into evidence-backed, economically justified, contract-bound solutions.

## Core Formula

```txt
Reality -> Evidence -> Diagnosis -> Money -> Offer -> Contract -> Authorized Execution
```

## Non-Negotiable Principles

### 1. Evidence Before Architecture

No solution is designed before evidence is captured, classified, and reviewed.

Evidence levels:

- `client_stated`
- `transcript_confirmed`
- `document_confirmed`
- `public_verified`
- `sector_inferred`
- `assumption`
- `unknown`

### 2. Pain Must Translate To Money

A pain that cannot be connected to time loss, revenue leakage, risk reduction, capacity increase, or operational cost is not ready for a paid solution.

### 3. The Simplest Solution That Captures Value Wins

Do not build an agentic workflow when a checklist, form, spreadsheet, reminder workflow, manual service, or lightweight process solves the business problem.

### 4. Offer Before Implementation

Every solution must be explainable, sellable, scoped, and economically justified before being converted into an SDD+ contract.

### 5. SDD+ Is The Contract System, Not The Imagination Engine

SDD+ formalizes justified decisions. It does not invent the business case.

SDD+ governs uncertainty.
Contracts govern execution.
Tests govern behavior.
Audits govern drift.
The human operator governs direction.

### 6. Execution Is Adapter-Based

The compiler must not be locked to any single model, vendor, coding assistant, orchestration platform, repo host, workflow tool, runtime, or cloud provider.

Tool examples are replaceable. Capability roles are stable.

### 7. RAKSO Is Separate From Vehicles

RAKSO is the operating system and compiler framework.

Vehicles are processed by RAKSO but must not become part of RAKSO's source of truth.

Vehicle-specific discovery, product scope, MVP definition, build plan, implementation detail, or customer evidence belongs in a separate vehicle repository or workspace.

### 8. Critical Lens Is Transversal

Every major artifact is audited for assumptions, overengineering, vendor lock-in, scope creep, missing evidence, operational fragility, commercial weakness, and repository-boundary violations.

### 9. Velocity Matters

A framework that cannot reduce internal friction cannot credibly promise external automation.

Each work cycle should produce a useful artifact, a closed decision, an audit result, or a clear stop reason.

## What This System Must Not Become

- A generic prompt library
- A hype-driven AI automation generator
- A vendor-specific skill system
- A single-tool implementation repo
- A lead research toy
- A proposal generator detached from evidence
- A custom app factory that defaults to overbuilding
- A vehicle-specific product repo
- A planning loop that never produces usable artifacts

## What This System Must Become

An internal operating system for transforming business reality into:

1. evidence ledger
2. operational diagnosis
3. pain-to-profit map
4. offer brief
5. SDD+ contract
6. authorized execution scope
7. audit result
8. next action

Future executable artifacts, deployment packages, agent profiles, workflows, integrations, scripts, schemas, runtime behavior, and product build remain unauthorized unless a later committed SDD+ contract explicitly permits them.
